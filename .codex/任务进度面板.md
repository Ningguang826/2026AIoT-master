下一步进入 SC171 开发板阶段：部署官方端侧 LLM fmodel，替换云端 DeepSeek API，实测板端本地模型推理后完整响应时延 `t`。

## 今日完成

- 修正 `系统响应时延 t` 的计时口径，使日志更接近手机手测：从用户真正停声开始，到首个非静音 PCM 写入播放后端结束。
- 将 TTS 首包前导静音裁掉，并把播放缓冲压到 30ms，减少“日志已写入但耳朵还没听到”的偏差。
- 保留稳定的 `cosyvoice-v1` websocket TTS 路径，移除不可用的旁支 TTS 尝试，避免测试指标被污染。
- 接入并收紧本地端点投机式 LLM 预取：可以隐藏一部分 ASR 服务端句尾等待，但最终文本新增关键约束时不会复用错误预取。
- 精简日志输出，只保留能直接指导时延判断和下一步决策的核心指标。
- 明确主机阶段结论：除云端 LLM 首 token 外，当前非模型链路已压到可作为板端基线的水平；继续优化重点转到开发板本地模型。

## 下一步

1. 在 SC171 开发板准备 license 与 fmodel，优先跑通官方 `fiboaisdk.api_nlp_py.NLPAPI.GenerateSync` 示例。
2. 实测 Qwen3-0.6B / deepseek-r1-1.5b 等候选模型单次生成耗时，先测短回复，再测约 30 字回复。

5. 在板端复测官方 `t`，重点区分模型推理耗时、TTS 首字耗时和音频设备耗时。
