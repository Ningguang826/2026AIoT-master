# 当前实现逻辑

## 1. 主线链路

当前主线是持续唤醒语音助手：

`短窗唤醒 -> 用户问题监听 -> DashScope ASR + 本地 VAD -> LLM -> DashScope TTS -> 连续 PCM 播放 -> 多轮追问`

主机阶段 LLM 仍使用 DeepSeek 云端流式接口，作为功能和时延对照链路。下一阶段在 SC171 开发板上用官方端侧 fmodel 替换云端 LLM API。

当前不再维护早期 `--input-text`、`--input-wav`、`--record-seconds`、`--realtime-seconds` 入口。

## 2. 唤醒与对话状态

`wake_loop.py` 负责持续唤醒状态机：

- `IdleListening`：短窗监听触发词。
- `ActivatedRecording`：命中触发词后播放“你好，我在。”。
- `Thinking`：拿到用户问题后请求 LLM。
- `Speaking`：流式 TTS 播放回复。
- `ErrorRecovery`：异常后回到可恢复状态。

触发词支持宽松匹配：`小智`、`小志`、`小知`、`小子`、`你好`、`hello`、`hi`、`开始对话`。

唤醒后不会每轮都退回待机。回答完成后继续听追问，连续 3 次空输入或无效输入才回到待机。

## 3. ASR 与本地端点

`streaming_voice.py` 负责麦克风输入、DashScope ASR 和本地 VAD 端点判断。

- 麦克风输入格式：`16kHz / mono / 16-bit PCM`。
- 默认 100ms 一帧读取音频，降低本地停声判断的量化误差。
- ASR 使用 DashScope `TranslationRecognizerChat`。
- 本地 VAD 使用 PCM RMS 判断语音活动。
- 正式问题监听默认开头静音上限 8 秒、尾部静音端点 0.6 秒、单轮硬上限 18 秒。

官方响应时延 `t` 的起点使用本地 VAD 记录的最后有效语音时间，而不是云端 ASR partial 文本到达时间。这样不会把“用户停声 -> ASR 服务端句尾确认”的等待隐藏掉，更接近手机手测口径。

## 4. 投机式 LLM 预取

为了减少 ASR 服务端句尾确认带来的等待，本地 VAD 判定用户停声后，会用当前最佳 ASR 文本提前启动 LLM。

投机预取只消费和缓存 LLM 文本，不占用播放设备，也不会在录音期间播 TTS。

最终 ASR 文本返回后再决定是否复用：

- 文本一致，或只多出“怎么样 / 吗 / 呢 / 啊”等语气尾巴时，复用预取流。
- 最终文本新增“公园 / 老人 / 室内”等实词约束时，丢弃预取，按最终文本重新请求 LLM。
- 太短或明显不完整的 ASR partial 不启动预取。

该策略不是消除 ASR 端点延迟，而是把一部分 ASR 等待和 LLM 首 token 等待重叠。

## 5. LLM

主机阶段 `llm_client.py` 使用 DeepSeek OpenAI-compatible 接口：

- 初始化时持久化 `OpenAI` client，减少重复创建客户端的开销。
- `stream_reply()` 使用流式输出。
- system prompt 要求回复简短、先给短开头，便于 TTS 尽快开始。
- 持续对话入口可复用投机预取流，并保留 LLM 启动时间和首 token 时间用于日志拆解。

当前主机阶段结论：云端 LLM 首 token 抖动仍是最大不确定项。下一阶段应在开发板上实测端侧 fmodel 的本地推理耗时。

## 6. TTS 与播放

主回答使用 `streaming_tts.py` 的 DashScope 真流式 TTS：

- TTS 模型：稳定使用 `cosyvoice-v1` / `longwan`。
- TTS 输出：22050Hz mono 16-bit PCM。
- 在 LLM 等待窗口内预热 TTS websocket，减少首片文本到首包音频的串行等待。
- 首片文本攒到约 4 个中文字符的可发音短语再提交，避免过短首片反而拉长首个有效 PCM 等待。
- TTS callback 收到 PCM 后立即入队。
- 播放线程按 RMS 裁掉首包前导静音。
- 播放缓冲当前为 30ms。

`系统响应时延 t` 的终点不是 TTS 首包到达，也不是普通 `waveOutWrite` 入队，而是首个非静音 PCM 写入播放后端的时间。

Windows 播放优先使用原生 `waveOut`，失败时降级 PyAudio。Linux / SC171 预留 `aplay -t raw ... stdin` 连续播放路径，并复用 `audio_output.py` 的声卡查找和 mixer 配置。

固定提示音“你好，我在。”和“没听清”走 `prompt_audio_cache.py`，首次生成 wav 缓存，后续直接播放。

## 7. 音频互斥

`audio_session.py` 提供最小音频会话锁：

- `IDLE`
- `RECORDING`
- `SPEAKING`

录音和播放互斥，播放期间不会启动麦克风监听，避免 TTS 被麦克风重新识别。投机式 LLM 预取只走网络文本流，不进入 `SPEAKING` 状态，也不写音频设备。

## 8. 日志与指标

INFO 日志只保留与比赛指标和决策直接相关的内容：

- `系统响应时延 t`
- `ASR 端点确认延迟`
- `LLM 首 token 延迟`
- `TTS 首片到首个非静音 PCM`
- 投机预取命中或丢弃

TTS 分片、前导静音裁剪、首包、首次写入、websocket 预热等内部诊断降到 DEBUG，避免干扰测试读数。

## 9. 板端迁移关注点

下一阶段在 SC171 开发板上重点验证：

- `fiboaisdk.api_nlp_py.NLPAPI.GenerateSync` 能否跑通。
- fmodel 单次生成耗时，尤其是短回复和约 30 字回复。
- 端侧模型是否支持流式或增量输出。
- 若仅支持非流式，先采用“整段生成完 -> 分句送 TTS”的方式接入主链路。
- 板端麦克风是否稳定输出 `16kHz / mono / 16-bit PCM`。
- `aplay -t raw -r 22050 -f S16_LE -c 1` 和声卡 mixer 是否稳定。
- 替换云端 LLM 后重新测试官方 `t`，拆分模型推理、TTS 首字、音频播放三部分耗时。
