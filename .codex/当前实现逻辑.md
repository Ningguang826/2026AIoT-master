# 当前实现逻辑

## 1. 主线链路

当前主线只保留持续唤醒语音助手：

`待机短窗监听 -> 触发词命中 -> 确认提示 -> 用户问题监听 -> DashScope ASR -> DeepSeek 流式回复 -> DashScope 真流式 TTS -> 连续 PCM 播放 -> 继续追问`

当前不再维护早期 `--input-text`、`--input-wav`、`--record-seconds`、`--realtime-seconds` 入口。

## 2. 麦克风与 ASR

- `wake_loop.py` 负责状态机，`streaming_voice.py` 负责实际麦克风和 ASR。
- 麦克风输入使用 `pyaudio` 读取 `16kHz / mono / 16-bit PCM`。
- 音频帧实时发送给 DashScope `TranslationRecognizerChat`。
- ASR callback 收到中间文本后，会把“云端已吐字”视为用户已开口，避免轻声或本地 RMS 阈值没命中时被开头静音截断。
- ASR callback 收到 `is_sentence_end` 后，优先按云端句尾收口。
- 如果云端没有及时给句尾，则使用本地 VAD 的最后语音时间和 `wake-end-silence-seconds` 做兜底。

## 3. VAD 与端点

当前 VAD 是轻量本地能量检测：

- 默认用 PCM RMS 阈值判断是否有语音活动。
- `question_initial_silence_seconds = 8.0`：唤醒后最多等用户开口 8 秒。
- `question_end_silence_seconds = 1.0`：检测到用户说话后，本地静音 1 秒作为输入结束兜底。
- `question_seconds = 18`：单次正式问题监听硬上限。

比赛响应指标 `t` 的起点：

- 如果是云端主动句尾：使用云端句尾时间。
- 如果是本地静音兜底：使用最后一次本地有效语音时间。

这样不会把本地等待尾静音的时间隐藏掉，符合“用户语句结束到 TTS 第一个字”的比赛口径。

## 4. 唤醒状态机

`WakeLoop` 使用同步单线程状态机：

- `IdleListening`：短窗 ASR 等触发词。
- `ActivatedRecording`：命中触发词后播放“你好，我在。”。
- `Thinking`：拿到用户问题后开始请求 DeepSeek。
- `Speaking`：流式 TTS 播放回复。
- `ErrorRecovery`：异常后回到可恢复状态。

触发词支持宽松匹配：

- `小智`
- `小志`
- `小知`
- `小子`
- `你好`
- `hello`
- `hi`
- `开始对话`

唤醒后不会每轮都退回待机。回答完成后继续听追问，连续 3 次空输入或无效输入才回到待机。

## 5. 不完整输入续听

为了兼顾 1 秒端点和用户思考停顿，当前不会用长尾静音硬等用户。

如果 ASR 返回类似：

- `我想问一下`
- `还有没有一些`
- `那室外的呢`
- `有哪`

系统会判定为疑似未说完，立即再开一个补充监听窗口，将两段文本合并后再送给 LLM。

这解决了 1 秒端点下容易把“开场白”直接送给 LLM 的问题。

## 6. 对话上下文

一次唤醒内维护短期上下文：

- 每轮成功回答后追加 user / assistant 消息。
- 当前保留最近 10 条消息。
- 回到待机后清空上下文。

这让“那室外的呢”“有没有室内的推荐”这类追问可以继承上一轮城市、对象和约束。

## 7. LLM

`llm_client.py` 使用 DeepSeek OpenAI-compatible 接口：

- 初始化时持久化 `OpenAI` client，减少每轮重复创建客户端的开销。
- `stream_reply()` 使用流式输出。
- system prompt 要求回复简短、先给短开头，便于 TTS 尽快开始。

当前主机侧云端测试显示：

- DeepSeek 首 token 延迟约 2 到 4 秒，是系统响应时延的主要瓶颈。
- 这属于当前云端阶段的可接受限制。
- 后续端侧部署小模型后，再把 `t <= 2s` 作为重点压测指标。

## 8. TTS 与播放

主回答使用 `streaming_tts.py` 真流式 TTS：

- LLM 文本增量进入分段缓冲。
- 达到短句、标点或长度阈值后调用 DashScope `SpeechSynthesizer.streaming_call()`。
- DashScope TTS callback 的 `on_data()` 收到 PCM 音频块后立即入队。
- 播放线程从队列读取 PCM，连续写入输出设备。

Windows：

- 优先使用原生 `waveOut` 队列播放 PCM。
- 若 `waveOut` 打开失败，再降级 PyAudio 输出。

Linux / SC171：

- 使用 `aplay -t raw ... stdin` 持续写入 PCM。
- 复用 `audio_output.py` 的声卡查找和 mixer 配置。

固定提示音：

- “你好，我在。”和“没听清”提示走 `prompt_audio_cache.py`。
- 首次生成 wav 缓存，后续直接播放，避免每次提示都走在线 TTS。

## 9. 音频资源互斥

`audio_session.py` 提供最小音频会话锁：

- `IDLE`
- `RECORDING`
- `SPEAKING`

录音和播放互斥，播放期间不会启动麦克风监听，避免 TTS 被麦克风重新识别。

- 单轮唤醒对话通过。
- 连续上下文可保持成都主题，前两轮为及格响应，第三轮因 DeepSeek 首 token 较慢导致不及格。
- 不完整输入续听合并已生效：`我想问一下` 后续听并合并为完整问题。
