# 开发计划

## 1. 当前定位

当前项目聚焦持续唤醒语音助手，不再扩展旧 GUI、视觉、多模态或已删除的早期 CLI 调试链路。

当前已稳定的主线：

`短窗唤醒 -> 麦克风实时输入 -> DashScope ASR -> 流式 LLM -> DashScope 真流式 TTS -> 连续 PCM 播放 -> 多轮追问`

第二阶段主机侧云端链路已完成。下一阶段优先在 Windows 主机上接入本地 Ollama 模型，验证端侧小模型路线能否满足比赛响应时延，再迁移到 SC171 板端。

## 2. 最近完成

- 持续唤醒状态机已跑通：待机、唤醒、录音、思考、播报、继续追问。
- 正式问题监听已改为 1 秒句尾静音，云端句尾优先，本地 VAD 兜底。
- 不完整输入续听合并已生效，避免“我想问一下”被直接送给 LLM。
- 一次唤醒内保留短期上下文，连续追问可以继承上一轮地点和约束。
- DashScope 真流式 TTS 已接入，Windows 使用 waveOut 连续播放，Linux / SC171 预留 `aplay` raw PCM。
- 官方响应时延指标已接入：`系统响应时延 t=x.xxxs，等级=优秀/及格/不及格`。
- DeepSeek 客户端已持久化，云端阶段只做轻量优化。
- 主机侧测试脚本已修正为当前竞赛口径。

## 3. 测试结论

最新主机侧云端日志结论：

- 功能链路：通过。
- 上下文连续性：基本通过。
- 空输入回待机：通过。
- 停顿防截断：续听合并已生效。
- 官方响应时延：云端 DeepSeek 多数为 3 到 5 秒，主要瓶颈是首 token。

因此第二阶段视为完成。云端 `t <= 2s` 不再作为当前阻塞，后续用本地 Ollama 和板端小模型继续压测。

## 4. 第三阶段计划：主机侧 Ollama 本地模型

目标：

在 Windows 主机上把 LLM 从云端 DeepSeek 切到本地 Ollama，保持 ASR、TTS、唤醒、音频互斥链路不变，验证本地模型首 token 和官方响应时延。

实施任务：

1. 新增本地 LLM 适配器，支持 Ollama 流式输出。
2. 抽象当前 `DeepSeekClient` 调用边界，避免 `wake_loop.py` 感知具体模型来源。
3. 在配置中新增：
   - `LLM_PROVIDER=deepseek|ollama`
   - `OLLAMA_BASE_URL=http://127.0.0.1:11434`
   - `OLLAMA_MODEL=<本地模型名>`
4. 复用现有 `answer_text()`、短期 history、TTS 分段和官方时延统计。
5. 跑主机侧完整测试：
   - 单轮唤醒
   - 连续 3 轮追问
   - 停顿防截断
   - 空输入回待机
   - 常驻演示

验收标准：

- Ollama 链路可完整回答并流式播报。
- 追问上下文不明显跑偏。
- 多数简单问答官方响应时延接近或低于 2 秒。
- 如果 Ollama 模型质量不足，保留 DeepSeek 作为云端降级。

## 5. 第四阶段计划：SC171 板端联调

进入条件：

- 主机侧 Ollama 链路完成 1 轮和 3 轮测试。
- 官方响应时延和回答质量达到可演示水平。

板端任务：

1. 验证麦克风输入是否稳定输出 `16kHz / mono / 16-bit PCM`。
2. 验证 `aplay -t raw -r 22050 -f S16_LE -c 1` 连续播放 PCM。
3. 检查 `audio_output.py` 声卡索引和 mixer 配置是否适配 SC171。
4. 迁移本地模型或端侧小模型推理链路。
5. 重新跑官方响应时延测试，并在作品中显示 `t`。


## 7. 当前决策

- 不再围绕早期已删除入口更新计划。
- 不在云端 DeepSeek 上继续强行压 `t <= 2s`。
- 下一步只聚焦 Ollama 本地模型适配和主机侧本地模型验收。
- Ollama 通过后再进入板端，不提前在板端排查大范围问题。
