# 开发计划

## 1. 当前定位

当前项目聚焦持续唤醒语音助手。主机侧云端链路已经完成基础功能和关键时延优化，新增了固定提示音缓存与即时应答词缓存，下一阶段转入 SC171 开发板端侧 LLM 部署调试。

当前主线链路：

`短窗唤醒 -> 麦克风实时输入 -> DashScope ASR -> 本地 VAD 端点 -> 投机式 LLM 预取 -> DeepSeek 流式回复 -> DashScope 真流式 TTS -> 连续 PCM 播放 -> 多轮追问`

主机阶段保留云端 DeepSeek 作为对照链路；板端阶段目标是用官方端侧 fmodel 替换云端 LLM API。

## 2. 今天完成的主机侧优化

### 计时口径

- `系统响应时延 t` 的起点改为本地 VAD 记录的最后有效语音时间，避免被云端 ASR partial 到达时间污染。
- `系统响应时延 t` 的终点改为首个非静音 PCM 写入播放后端，更接近用户实际听到第一个字的时刻。
- 日志保留核心拆解项：`ASR 端点确认延迟`、`LLM 首 token 延迟`、`TTS 首片到首个非静音 PCM`。

### TTS 与播放

- 保留稳定 `cosyvoice-v1` websocket 路径。
- 在 LLM 等待窗口内预热 TTS websocket，减少首片文本到首包音频的串行等待。
- 首片文本不再一个字就提交，而是攒到可发音短语，降低 TTS 服务端等待上下文的概率。
- 播放线程按 RMS 裁掉 TTS 首包前导静音。
- 播放缓冲压到 30ms，在首字延迟和播放稳定性之间取当前主机侧折中。

### 固定提示音缓存与即时应答词缓存

- 唤醒确认和未听清兜底短语首次生成 wav 缓存，后续直接播放。
- 启动时预热常用短语，避免第一轮额外在线等待。
- 即时应答词不再当作普通文本送给 DashScope 流式 TTS，而是先加载本地缓存 wav，解析 raw PCM 后直接灌入同一条播放队列。
- 应答词后补一段 `ack_gap_ms` 静音停顿，再接正文流式 TTS，兼顾尽快出声和听感自然。
- CLI 未显式传 `--ack-words` 时，默认回退到 `WakeLoopConfig` 内置整句口语词表；`--no-ack-word` 可关闭做时延对照。

### ASR 端点与 LLM 预取

- 正式问题监听使用 100ms 音频帧和 0.6s 本地静音端点。
- 本地 VAD 判定停声后，会用当前最佳 ASR 文本投机启动 LLM，把 ASR 服务端句尾等待和 LLM 首 token 等待重叠。
- 投机复用规则已收紧：最终 ASR 文本新增实词约束时丢弃预取，避免为了抢时间忽视“公园”等关键词。

## 3. 主机阶段结论

- 功能链路已经可用：唤醒、连续追问、上下文、空输入回待机、续听合并、TTS 播放和官方 `t` 显示均已接入。
- 固定提示音缓存已经稳定；即时应答词缓存也已切到当前最合理的实现路径。
- 除 LLM 推理外，主机侧链路开销已经压到可作为板端基线的水平。
- 云端 DeepSeek 首 token 抖动仍是主机侧最大不确定项，继续压云端模型收益有限。
- 下一步应转向开发板端侧 LLM，用本地小模型替换云端 API 后重新评估完整 `t`。

## 4. 开发板阶段计划

目标：

在 SC171 上部署官方已量化端侧 LLM fmodel，跑通本地模型推理，并接入当前持续唤醒语音链路。

优先路线：

1. 准备板端 license 与 fmodel 文件。
2. 用官方示例跑通 `fiboaisdk.api_nlp_py.NLPAPI.GenerateSync`。
3. 实测候选模型推理耗时：
   - 短回复耗时。
   - 约 30 字回复耗时。
   - 是否支持流式或增量输出。
4. 新增 `FiboLocalLLMClient`：
   - 统一 prompt 模板。
   - 封装 `GenerateSync`。
   - 清洗模型输出。
   - 对上层暴露与现有 `stream_reply()` 兼容的接口。
5. 替换主链路中的 DeepSeek 调用，先跑通完整闭环。
6. 在板端复测官方 `t`，拆分模型推理、TTS 首字、音频播放三个主要耗时。

## 4.1 VSCode 远程连板调试流程（下一步执行）

依据官方 `软件平台使用/VScode远程连接.pdf`、`WIFI连接.pdf`、`案例部署包使用指南`。连接走 SSH + Remote-SSH，ADB 仅用于查 IP 与推文件。

固定事实：板端默认用户 `fibo` / 密码 `123`；端侧工作目录约定 `/home/fibo/project/<案例名>/`；license 固定在 `/home/fibo/qcom_6490_license/`（key1/2/3.pem + license.bin）。

阶段一：建立 VSCode 远程连接

1. 电脑与 SC171 接入同一 Wi-Fi（建议 5GHz，2.4G/休眠会导致 SSH 频繁掉线）。板端未联网时先用 Type-C + `adb shell ./usr/bin/fibo_wifi_api_test` 配网。
2. 查板端 IP：PC 端 `adb root` -> `adb shell` -> `ifconfig`，记下 wlan IP。
3. 板端装 SSH 服务：`apt update` -> `apt install openssh-server`。
4. PC 端 VSCode 装 Remote-SSH 扩展 -> `ssh fibo@<板端IP>` -> 选 Linux -> 密码 `123`。
5. 板端执行一次防掉线：`systemctl mask sleep.target suspend.target hibernate.target hybrid-sleep.target`。
6. 远端装 Python 扩展，`Python: Select Interpreter` 选板端 Python（官方示例 3.8.10）。
7. 连接判据：能打开 `/home/fibo/`、开集成终端、跑通一个 `hello.py`。

阶段二：落地端侧 LLM 文件并验证环境

1. adb 推 Qwen3-0.6B（fmodel + `qwen3-0.6b.py`）到板上，或打 zip 后 `adb push` + `adb shell unzip -d /home/fibo/`。
2. 连上后第一时间确认三项（决定能否跑通）：
   - `ls /home/fibo/qcom_6490_license/` 四个 license 文件是否齐全；
   - `python3 -c "import fiboaisdk; print('ok')"` SDK 是否预装；
   - 模型实际落盘路径，对齐脚本里 `api.Init(...)` 第一参（示例 `/home/fibo/qwen3/qwen3`，需改成真实路径）。

阶段三：跑通官方示例 + 实测时延

1. 远端终端跑 `qwen3-0.6b.py`，确认 `license_api.Init` 返回 0、`GenerateSync` 能出文本。
2. 加计时测三档：短回复、约 30 字回复、是否有流式/增量接口（判断非流式是否拖垮 `t`）。
3. 顺带验证 ASR 麦克风（`arecord -l`）与 `aplay` raw PCM 播放，为接主链路铺路。

## 5. 板端优先验证项

- 麦克风输入是否稳定满足 `16kHz / mono / 16-bit PCM`。
- `aplay -t raw -r 22050 -f S16_LE -c 1` 是否能稳定连续播放。
- 声卡 mixer 配置是否与 `audio_output.py` 当前 Linux 分支匹配。
- fmodel 推理单次耗时是否满足作品要求。
- 端侧模型回答是否足够短、稳、可控，必要时通过短 system prompt 和 few-shot 约束。
- Ctrl+C 或异常打断时，`aplay` 连续播放是否残留明显尾音。

## 7. 当前决策

- 不再继续围绕云端 LLM 强行压到最终指标。
- 主机侧后续只作为功能和时延对照，不再引入额外旁支模型或 TTS 后端。
- 固定提示音缓存保持当前 wav 缓存方案；即时应答词保持当前“本地缓存 wav -> raw PCM -> 同队列预灌”的方案。
- 开发重点转到 SC171 板端：先跑通本地模型，再接入主链路，再复测官方 `t`。
