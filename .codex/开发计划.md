# 开发计划

## 1. 当前定位

当前项目聚焦持续唤醒语音助手。主机侧云端链路已经完成基础功能和关键时延优化，下一阶段转入 SC171 开发板端侧 LLM 部署调试。

当前主线链路：

`短窗唤醒 -> 麦克风实时输入 -> DashScope ASR -> LLM -> DashScope 真流式 TTS -> 连续 PCM 播放 -> 多轮追问`

主机阶段保留云端 DeepSeek 作为对照链路；板端阶段目标是用官方端侧 fmodel 替换云端 LLM API。

## 2. 今天完成的主机侧优化

### 计时口径

- `系统响应时延 t` 的起点改为本地 VAD 记录的最后有效语音时间，避免被云端 ASR partial 到达时间污染。
- `系统响应时延 t` 的终点改为首个非静音 PCM 写入播放后端，更接近用户实际听到第一个字的时刻。
- 日志保留核心拆解项：`ASR 端点确认延迟`、`LLM 首 token 延迟`、`TTS 首片到首个非静音 PCM`。

### TTS 与播放

- 保留稳定 `cosyvoice-v1` websocket 路径。
- 在 LLM 等待窗口内预热 TTS websocket，减少首片文本到首包音频的串行等待。
- 首片文本不再一个字就提交，而是攒到可发音短语，降低 TTS 服务端等待上下文的概率。
- 播放线程按 RMS 裁掉 TTS 首包前导静音。
- 播放缓冲压到 30ms，在首字延迟和播放稳定性之间取当前主机侧折中。

### ASR 端点与 LLM 预取

- 正式问题监听使用 100ms 音频帧和 0.6s 本地静音端点。
- 本地 VAD 判定停声后，会用当前最佳 ASR 文本投机启动 LLM，把 ASR 服务端句尾等待和 LLM 首 token 等待重叠。
- 投机复用规则已收紧：最终 ASR 文本新增实词约束时丢弃预取，避免为了抢时间忽视“公园”等关键词。

## 3. 主机阶段结论

- 功能链路已经可用：唤醒、连续追问、上下文、空输入回待机、续听合并、TTS 播放和官方 `t` 显示均已接入。
- 除 LLM 推理外，主机侧链路开销已经压到可作为板端基线的水平。
- 云端 DeepSeek 首 token 抖动仍是主机侧最大不确定项，继续压云端模型收益有限。
- 下一步应转向开发板端侧 LLM，用本地小模型替换云端 API 后重新评估完整 `t`。

## 4. 开发板阶段计划

目标：

在 SC171 上部署官方已量化端侧 LLM fmodel，跑通本地模型推理，并接入当前持续唤醒语音链路。

优先路线：

1. 准备板端 license 与 fmodel 文件。
2. 用官方示例跑通 `fiboaisdk.api_nlp_py.NLPAPI.GenerateSync`。
3. 实测候选模型推理耗时：
   - 短回复耗时。
   - 约 30 字回复耗时。
   - 是否支持流式或增量输出。
4. 新增 `FiboLocalLLMClient`：
   - 统一 prompt 模板。
   - 封装 `GenerateSync`。
   - 清洗模型输出。
   - 对上层暴露与现有 `stream_reply()` 兼容的接口。
5. 替换主链路中的 DeepSeek 调用，先跑通完整闭环。
6. 在板端复测官方 `t`，拆分模型推理、TTS 首字、音频播放三个主要耗时。

## 5. 板端优先验证项

- 麦克风输入是否稳定满足 `16kHz / mono / 16-bit PCM`。
- `aplay -t raw -r 22050 -f S16_LE -c 1` 是否能稳定连续播放。
- 声卡 mixer 配置是否与 `audio_output.py` 当前 Linux 分支匹配。
- fmodel 推理单次耗时是否满足作品要求。
- 端侧模型回答是否足够短、稳、可控，必要时通过短 system prompt 和 few-shot 约束。

## 6. 当前测试命令

主机侧静态检查：

```powershell
python -m py_compile main_cli.py wake_loop.py streaming_voice.py streaming_tts.py audio_session.py prompt_audio_cache.py llm_client.py audio_output.py settings.py
```

主机侧持续唤醒对照：

```powershell
python main_cli.py --wake-loop --wake-rounds 1
python main_cli.py --wake-loop --wake-rounds 3
```

## 7. 当前决策

- 不再继续围绕云端 LLM 强行压到最终指标。
- 主机侧后续只作为功能和时延对照，不再引入额外旁支模型或 TTS 后端。
- 开发重点转到 SC171 板端：先跑通本地模型，再接入主链路，再复测官方 `t`。
